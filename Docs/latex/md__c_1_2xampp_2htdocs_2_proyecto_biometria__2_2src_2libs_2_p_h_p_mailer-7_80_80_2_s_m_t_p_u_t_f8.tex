For most of its existence, SMTP has been a 7-\/bit channel, only supporting US-\/\+ASCII characters. This has been a problem for many languages, especially those that use non-\/\+Latin scripts, and has led to the development of various workarounds.

The first major improvement, introduced in 1994 in {\texttt{RFC 1652}} and extended in 2011 in {\texttt{RFC 6152}}, was the addition of the {\ttfamily 8BITMIME} SMTP extension, which allowed raw 8-\/bit data to be included in message bodies sent over SMTP. This allowed the message {\itshape contents} to contain 8-\/bit data, including things like UTF-\/8 text, even though the SMTP protocol itself was still firmly 7-\/bit. This worked by having the server switch to 8-\/bit after the headers, and then back to 7-\/bit after the completion of a {\ttfamily DATA} command.

From 1996, messages could support {\texttt{RFC 2047 encoding}}, which permitted inserting characters from any character set into header {\itshape values} (but not names), but only by encoding them in somewhat unreadable ways to allow them to survive passage through a 7-\/bit channel. \end{DoxyCode}


Here the accented {\ttfamily Ã¶} is encoded as {\ttfamily =C3=B6}, which is the UTF-\/8 encoding of the 2-\/byte character, and the whole thing is wrapped in {\ttfamily =?utf-\/8?Q?} to indicate that it uses the UTF-\/8 charset and {\ttfamily quoted-\/printable} encoding. This is a bit of a hack, and not very human-\/friendly, but it works.

Similarly, 8-\/bit message bodies could be encoded using the same {\ttfamily quoted-\/printable} and {\ttfamily base64} content transfer encoding (CTE) schemes, which preserved the 8-\/bit content while encoding it in a format that could survive transmission through a 7-\/bit channel.

Domain names were originally also stuck in a 7-\/bit world, actually even more constrained to only a subset of the US-\/\+ASCII character set. But of course, many people want to have domains in their own language/script. Internationalized domain name (IDN) permitted this, using yet another complex encoding scheme called punycode, defined for domain names in 2003 in {\texttt{RFC 3492}}. This finally allowed the domain part (after the {\ttfamily @}) of email addresses to contain UTF-\/8, though it was actually an illusion preserved by email client applications. For example, an address of {\ttfamily user@cafÃ©.\+example.\+com} translates to {\ttfamily user@xn-\/-\/caf-\/dma.\+example.\+com} in punycode, rendering it mostly unreadable, but 7-\/bit friendly, and remaining compatible with email clients that don\textquotesingle{}t know about IDN.

The one remaining part of email that could not handle UTF-\/8 is the local part of email addresses (the part before the {\ttfamily @}).

I\textquotesingle{}ve only mentioned UTF-\/8 here, but most of these approaches also allowed other character sets that were popular, such as {\texttt{the ISO-\/8859 family}}. However, UTF-\/8 solves so many problems that these other character sets are gradually falling out of favour, as UTF-\/8 can support all languages.

{\ttfamily SMTPUTF8} is another SMTP extension, defined in {\texttt{RFC 6531}} in 2012. This essentially solves the whole problem, allowing the entire SMTP conversation â€” commands, headers, and message bodies â€” to be sent in raw, unencoded UTF-\/8.

But there\textquotesingle{}s a problem with this approach\+: adoption. If you send a UTF-\/8 message to a recipient whose mail server doesn\textquotesingle{}t support this format, the sender has to somehow downgrade the message to make it survive a transition to 7-\/bit. This is a hard problem to solve, especially since there is no way to make a 7-\/bit system support UTF-\/8 in the local parts of addresses. Several other PHP email libraries have implemented a halfway solution to {\ttfamily SMTPUTF8}, adding only the ability to support UTF-\/8 in email addresses, not elsewhere in the protocol. I wanted \doxyref{PHPMailer}{p.}{namespace_p_h_p_mailer} to do it "{}the right way"{}, and this has taken much longer. \doxyref{PHPMailer}{p.}{namespace_p_h_p_mailer} now supports UTF-\/8 everywhere, and does not need to use transfer or header encodings for UTF-\/8 text when connecting to an {\ttfamily SMTPUTF8}-\/capable mail server.

This support is handled automatically\+: if you add an email address that requires UTF-\/8, \doxyref{PHPMailer}{p.}{namespace_p_h_p_mailer} will use UTF-\/8 for everything. If not, it will fall back to 7-\/bit and encode the message as necessary.

The one place you will need to be careful is in the selection of the address validator. By default, \doxyref{PHPMailer}{p.}{namespace_p_h_p_mailer} uses PHP\textquotesingle{}s built-\/in {\ttfamily filter\+\_\+var} validator, which does not allow UTF-\/8 email addresses. When \doxyref{PHPMailer}{p.}{namespace_p_h_p_mailer} spots that you have submitted a UTF-\/8 address, but have not altered the default validator, it will automatically switch to using a UTF-\/8-\/compatible validator. As soon as you do this, any SMTP connection you make will {\itshape require} that the server you connect to supports {\ttfamily SMTPUTF8}.
